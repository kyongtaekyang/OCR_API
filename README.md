# OCR_API

Writing Image Benchmark System

AI-based benchmark system comparing `qwen2.5vl:latest` and `gemma4:latest` on EFL student writing image evaluation tasks.

## Project Purpose

Compare two Ollama vision-language models on:
- Writing image analysis accuracy
- JSON output schema compliance
- Response speed and stability
- API and AWS deployment readiness

## Project Structure

```
C:\Project\writing_api
├─ config\               # Model and benchmark configuration
├─ data\
│  ├─ input_images\      # Writing image files (.jpg/.png)
│  ├─ prompts\           # system_prompt.txt, output_prompt.txt, combined_eval_prompt.txt
│  └─ expected_outputs\  # Ground-truth JSON files
├─ results\

```

1. Place writing image files in `data/input_images/` (`.jpg`, `.jpeg`, `.png` supported)
2. Place expected output JSON in `data/expected_outputs/`
3. Review `data/prompts/system_prompt.txt` and `data/prompts/output_prompt.txt`

## Prompt Files

### system_prompt.txt
Defines the assessment rules:
- Image reading (handwriting only, ignore printed text)
- Error tagging: `[G]` Grammar, `[V]` Vocabulary, `[O]` Word Order, `[P]` Punctuation, `[S]` Spelling, `[C]` Coherence
- Scoring schema for Keystone and Zoom courses
- Mandatory self-check rules

### output_prompt.txt
Defines the output JSON structure with example scores.  
Note: Example scores are for structure reference only — not ground truth.

### combined_eval_prompt.txt
Auto-generated by `prompt_builder.py` from the above two files.
Includes all constraints for valid JSON output.

## Model Runtime Policy

Model-specific runtime settings live in `config/models.json`.

| Model | Timeout | Retries | Operational policy |
|---|---:|---:|---|
| `qwen2.5vl:latest` | 900s | 0 | Large VLM retest mode: allow one long inference attempt and avoid repeating an expensive failed call. |
| `gemma4:latest` | 300s | 1 | Current API/AWS candidate: keep the existing bounded retry policy. |

The benchmark summary records the applied `timeout_seconds`, `max_retries`, request `options`, `error`, and `response_size_chars` for each model. If `qwen2.5vl:latest` still fails under the extended policy, treat it as unsuitable for the current local API path until hardware, model loading, or prompt length is adjusted.

## Benchmark Metrics

| Metric | Description |
|---|---|
| `json_parse_success` | Whether model output parsed as valid JSON |
| `schema_compliance_score` | Weighted score of required key and structure completeness |
| `required_key_completeness` | Fraction of required top-level keys present |
| `score_math_valid` | Whether subtotals equal sum of sub-scores |
| `score_anomalies` | List of scores exceeding max_score |
| `original_writing_similarity` | SequenceMatcher similarity (0–1) |
| `corrected_writing_similarity` | SequenceMatcher similarity (0–1) |
| `error_tag_f1` | F1 score of error tag category counts |
| `total_score_difference` | Absolute difference in total score |
| `overall_accuracy_score` | Weighted composite (see below) |
| `api_readiness_score` | API deployment suitability score |
| `duration_seconds` | Model inference time |

### overall_accuracy_score Weights

| Component | Weight |
|---|---|
| JSON schema compliance | 20% |
| original_writing similarity | 20% |
| corrected_writing similarity | 20% |
| error tag F1 | 20% |
| score similarity | 10% |
| error_analysis similarity | 10% |

## Schema Compatibility

The system auto-detects which schema a JSON output follows:
- **system_prompt_schema**: `grammar.sentence_accuracy`, `grammar.article_preposition`
- **output_prompt_schema**: `grammar.subject_verb_agreement`, `grammar.articles_prepositions`

The comparator uses common fields (subtotals, total_score) when schemas differ.

## Score Anomaly Detection

During validation, the system checks:
1. `score > max_score` for any scoring field → anomaly recorded
2. `subtotal != sum of sub-scores` → anomaly recorded

Anomalies appear in `comparison.score_anomalies` and are counted in `api_readiness_score`.

## API Readiness Evaluation

| Component | Weight |
|---|---|
| JSON validity | 25% |
| Response time | 20% |
| Key completeness (consistency) | 15% |
| Error handling (no timeout/failure) | 15% |
| Output schema compliance | 15% |
| Score math validity | 10% |
| Strict JSON penalty | -10% if output not pure JSON |

## AWS Deployment Considerations

- **Lambda**: Not suitable for model inference (memory/GPU/timeout limits). Use for orchestration only.
- **ECS on EC2 GPU / EC2 GPU**: Recommended for Ollama container serving.
- **S3**: Ideal for input image storage and result archiving.
- **CloudWatch**: For logs and metrics monitoring.
- **Cold start**: Ollama model loading time must be accounted for.

## Troubleshooting

| Problem | Solution |
|---|---|
| `ollama_server_reachable: FAIL` | Start Ollama: `ollama serve` |
| `model_qwen2.5vl: FAIL` | Run `ollama pull qwen2.5vl:latest` |
| `JSONDecodeError` in parse | Model returned non-JSON output — check raw_outputs/ |
| Timeout | Increase `timeout_seconds` in `config/benchmark_config.json` |
| `qwen2.5vl:latest` takes too long or fails | Review `config/models.json`; qwen uses a 900s single-attempt retest policy to avoid duplicate long failures |
| Image not found | Check path; supported formats: jpg, jpeg, png, bmp, gif, webp |

## Model Test APIs

This project supports API-style writing analysis for two local Ollama model aliases:

- `qwen2` -> `qwen2.5vl:latest`
- `gemma` -> `gemma4:latest`

The current executable test path uses the local Ollama models on this PC. The AWS S3 artifact plus GPU inference runtime design is tracked as a deployment TODO, not a completed runtime requirement for local testing.

Architecture:

```text
Client
  -> FastAPI /analyze-writing/qwen2 or /analyze-writing/gemma
  -> OCR JSON generation or validation
  -> prompt build from system_prompt.txt + output_prompt.txt + OCR JSON
  -> local Ollama model endpoint
  -> raw output save
  -> JSON parsing, validation, optional comparison
  -> benchmark summary and API response JSON
```

### Local Test Runtime

- `config/models.json` maps `qwen2` and `gemma` to local Ollama model names.
- `/analyze-writing/qwen2` forces `model_names=["qwen2"]` and `inference_mode=local`.
- `/analyze-writing/gemma` forces `model_names=["gemma"]` and `inference_mode=local`.
- `/test-page` provides a browser page for uploading OCR JSON or an image and running the API.
- AWS remote payload support remains in code for later deployment work.
- Prompt files are preserved: the API reads `data/prompts/system_prompt.txt` and `data/prompts/output_prompt.txt` and appends OCR context after the original prompt text.

### OCR JSON Schema

```json
{
  "metadata": {
    "class": "string or not provided",
    "level": "string or not provided",
    "title": "string or not provided",
    "topic": "string or not provided",
    "writing_type": "string or not provided"
  },
  "handwritten_text": "full text from student handwriting",
  "lines": [
    {"line_no": 1, "text": "string", "confidence": 0.0}
  ],
  "source": {
    "type": "image|json",
    "filename": "string",
    "ocr_engine": "manual|mock|textract|external"
  }
}
```

Missing metadata fields default to `not provided`. `confidence` must be between `0` and `1`. Image OCR currently uses a mock adapter so the structure is ready for AWS Textract, Google Vision, Azure OCR, or Tesseract.

### Security and Reliability

- Allowed uploads: images `.jpg`, `.jpeg`, `.png`, `.webp`; JSON `.json`; prompts `.txt`.
- Upload size defaults to `MAX_UPLOAD_MB=20`.
- Uploaded filenames are not used for result paths; temporary safe filenames and sanitized run names are used.
- Path traversal is blocked by result path helpers.
- API keys and secrets must never be hardcoded or logged.
- Remote calls use timeout, retry, structured errors, and status-code categories.
- JSON parse failures do not crash the run; raw output is saved and parse errors are returned.

### AWS Deployment Notes

- Recommended serving layers: ECS on EC2 GPU, EC2 GPU, EKS GPU nodes, or SageMaker endpoint.
- API Gateway + Lambda is recommended only for orchestration, not model inference.
- Large VLM inference should run in a GPU runtime with model artifacts stored in S3, cached on EBS/EFS, and logs sent to CloudWatch.
- Store uploaded inputs and output artifacts in S3 for production retention.
- Store endpoint API keys in Secrets Manager and inject them as environment variables.
